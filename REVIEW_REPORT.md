# Review Report

Дата проверки: 2026-06-10

Проверенный проект: `electronic-gradebook`

Команды проверки:

```bash
python3 -m unittest discover -s tests -v
.venv/bin/python -m unittest discover -s tests -v
```

Результат: все текущие тесты проходят, `7/7 OK`.

## 1. Логика проекта

### `src/gradebook.py`

**Проблема: `get_score_by_name()` молча возвращает `None`, если студент не найден.**

Файл: `src/gradebook.py`, строки 22-25.

Методы `get_score_by_id()`, `update_score()` и `delete_student()` явно выбрасывают `ValueError`, если студента нет. Но `get_score_by_name()` при отсутствии имени просто завершает цикл и возвращает `None`. Это создает несимметричное поведение API: вызывающий код не может надежно отличить "студент не найден" от ошибочного или пустого значения без дополнительной проверки.

Рекомендация: привести поведение к единому контракту, например выбрасывать `ValueError("Студент с таким именем не найден")`, либо явно задокументировать, что метод возвращает `None`.

**Проблема: имена студентов не уникальны, но поиск по имени возвращает только первого найденного.**

Файл: `src/gradebook.py`, строки 22-25.

`GradeBook` хранит студентов по `student_id`, но `get_score_by_name()` ищет по имени и возвращает оценку первого совпадения. Если добавить двух студентов с одинаковым именем, результат зависит от порядка вставки и может быть неожиданным.

Рекомендация: либо запретить дублирующиеся имена при добавлении, либо возвращать список совпадений, либо переименовать метод так, чтобы было ясно, что он возвращает первое совпадение.

**Проблема: оценка не валидируется.**

Файлы: `src/gradebook.py`, строки 7-15; `src/student.py`, строки 3-7.

Сейчас можно добавить или обновить студента с оценкой `-10`, `999`, строкой `"A"` или `None`. Тип `score: int` в dataclass является подсказкой, но не проверяется во время выполнения.

Рекомендация: определить допустимый диапазон оценок для проекта и проверять его в `add_student()` и `update_score()`. Если журнал должен работать только с числами, проверять тип значения.

**Проблема: входные данные студента не валидируются.**

Файлы: `src/gradebook.py`, строки 7-10; `src/student.py`, строки 3-7.

`student_id` и `name` принимаются без проверок. Возможны пустые имена, `None`, отрицательные id или id неподходящего типа. Это может привести к некорректному состоянию журнала.

Рекомендация: добавить проверки для `student_id` и `name`: id должен быть положительным идентификатором нужного типа, имя должно быть непустой строкой.

**Наблюдение: `show_all_students()` возвращает изменяемые объекты `Student`.**

Файл: `src/gradebook.py`, строки 27-28.

Метод возвращает список объектов из внутреннего словаря. Внешний код может изменить `student.score`, `student.name` или `student.student_id` напрямую, обходя методы `GradeBook` и их будущую валидацию.

Рекомендация: если нужна защита инвариантов, возвращать копии объектов, неизменяемые DTO или предоставить отдельные методы чтения.

## 2. Стиль кода

### `main.py`

**Проблема: wildcard import усложняет чтение и сопровождение.**

Файл: `main.py`, строка 2.

`from src.utils import *` скрывает, какие функции используются, и может создавать конфликты имен.

Рекомендация: заменить на явный импорт `from src.utils import print_students`.

### `src/utils.py`

**Проблема: неиспользуемый импорт.**

Файл: `src/utils.py`, строка 1.

`Student` импортируется, но не используется.

Рекомендация: удалить импорт или использовать его в type hints, если планируется типизировать `print_student(student: Student)`.

### Общий стиль

**Проблема: отсутствуют type hints в публичных методах `GradeBook`.**

Файл: `src/gradebook.py`, строки 4-33.

В `Student` типы указаны, но методы `GradeBook` и функции `utils` не типизированы. Из-за этого контракт методов приходится восстанавливать по реализации и тестам.

Рекомендация: добавить аннотации параметров и возвращаемых значений, например `student_id: int`, `name: str`, `score: int`, `-> None`, `-> int`, `-> list[Student]`.

**Проблема: форматирование частично не соответствует PEP 8.**

Файлы: `main.py`, `src/gradebook.py`, `src/student.py`, `src/utils.py`, `tests/test_gradebook.py`.

Между импортами и объявлениями классов/функций не всегда есть две пустые строки. Для маленького учебного проекта это не критично, но автоматический форматтер сделал бы стиль стабильнее.

Рекомендация: добавить `black`/`ruff` или хотя бы следовать PEP 8 вручную.

## 3. Тесты

### `tests/test_gradebook.py`

**Сильная сторона: базовые сценарии покрыты.**

Файл: `tests/test_gradebook.py`, строки 12-41.

Тесты проверяют добавление, обновление, поиск по id, поиск по имени, удаление, дубликат id и обновление неизвестного студента.

**Проблема: нет теста на поиск неизвестного имени.**

Файл: `tests/test_gradebook.py`, строки 24-26.

Сейчас поведение `get_score_by_name("Unknown")` не зафиксировано. Это скрывает проблему с неявным `None`.

Рекомендация: добавить тест на отсутствующее имя и выбрать ожидаемый контракт: `ValueError` или `None`.

**Проблема: нет тестов на некорректные оценки.**

Файл: `tests/test_gradebook.py`.

Не проверяются отрицательные оценки, оценки выше допустимого диапазона и значения неправильного типа.

Рекомендация: добавить тесты для `add_student()` и `update_score()` с невалидными значениями.

**Проблема: нет тестов на некорректные id и имена.**

Файл: `tests/test_gradebook.py`.

Не проверяются пустые имена, `None`, отрицательные id и id неправильного типа.

Рекомендация: добавить негативные тесты после определения правил валидации.

**Проблема: не тестируется порядок и содержимое `show_all_students()`.**

Файл: `tests/test_gradebook.py`, строки 12-14 и 28-31.

Текущие тесты проверяют только длину списка. Они не подтверждают, что возвращаются правильные студенты и что порядок вывода соответствует ожиданиям.

Рекомендация: проверять `student_id`, `name`, `score` возвращаемых объектов или использовать сравнение ожидаемых dataclass-экземпляров.

## 4. Структура проекта

**Проблема: нет файла с описанием проекта и инструкциями запуска.**

Файл отсутствует: `README.md`.

Без README неочевидно, как запускать приложение, тесты и какие правила у электронного журнала: диапазон оценок, уникальность имен, допустимые значения id.

Рекомендация: добавить README с кратким описанием, командами запуска и правилами доменной модели.

**Проблема: нет файла зависимостей или метаданных проекта.**

Файлы отсутствуют: `requirements.txt`, `pyproject.toml`.

Сейчас проект использует только стандартную библиотеку, но для воспроизводимости полезно явно указать версию Python и инструменты разработки, если появятся `ruff`, `black` или `pytest`.

Рекомендация: добавить `pyproject.toml` с минимальными настройками проекта, форматтера и линтера.

**Наблюдение: `.idea/` и `.venv/` игнорируются, но уже присутствуют в рабочей директории.**

Файл: `.gitignore`, строки 1-3.

Это нормально для локальной среды, но такие директории не должны попадать в репозиторий. Текущий `git status` чистый, значит они, вероятно, не отслеживаются.

Рекомендация: оставить их в `.gitignore`; при необходимости проверить `git ls-files .idea .venv`.

**Наблюдение: `src/` не содержит `__init__.py`.**

Папка: `src/`.

Импорты `from src.gradebook import GradeBook` работают в текущем запуске благодаря namespace packages и запуску из корня проекта. Но для простого учебного проекта явный `src/__init__.py` может сделать пакетную структуру понятнее.

Рекомендация: добавить `src/__init__.py`, если проект планируется развивать как пакет.

## 5. Возможные ошибки и риски

**Риск: некорректные данные могут попасть в состояние журнала.**

Файлы: `src/gradebook.py`, `src/student.py`.

Из-за отсутствия валидации можно создать студентов с пустыми именами, отрицательными id или невозможными оценками. Последующие функции будут работать с этими данными как с валидными.

Рекомендация: централизовать валидацию в `GradeBook` или в `Student.__post_init__()`.

**Риск: прямое изменение объектов из `show_all_students()` обходит бизнес-логику.**

Файл: `src/gradebook.py`, строки 27-28.

Пример риска: внешний код получает список студентов и меняет `students[0].score = -100`, минуя `update_score()`.

Рекомендация: возвращать копии или сделать `Student` неизменяемым через `@dataclass(frozen=True)` и обновлять записи заменой объекта.

**Риск: пользовательский интерфейс в `main.py` является демонстрационным, а не интерактивным.**

Файл: `main.py`, строки 4-17.

Программа создает заранее заданных студентов и печатает результат. Для реального электронного журнала нет ввода, сохранения данных, загрузки данных или обработки пользовательских команд.

Рекомендация: если проект должен быть приложением, отделить демонстрационный сценарий от CLI/API и добавить полноценный интерфейс работы с журналом.

**Риск: команда `python` не работает в текущем окружении.**

Проверка: `python -m unittest discover -s tests -v` завершилась ошибкой `zsh:1: command not found: python`.

При этом `python3 -m unittest discover -s tests -v` и `.venv/bin/python -m unittest discover -s tests -v` работают успешно.

Рекомендация: в документации использовать `python3` или путь к виртуальному окружению.

## Итог

Проект небольшой и понятный: базовая модель `GradeBook` работает, текущие тесты проходят. Главные зоны улучшения: явный контракт ошибок, валидация входных данных, расширение негативных тестов, явные импорты и документация запуска. Самый важный потенциальный дефект сейчас находится в `get_score_by_name()`: метод ведет себя иначе, чем остальные методы поиска и изменения, и не сообщает об отсутствии студента.
