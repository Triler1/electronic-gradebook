# Subagents Review

## Logic reviewer result

### Что проверял агент

`logic_reviewer` выполнил read-only проверку логики проекта: `GradeBook`, модель `Student`, утилиты вывода, основной сценарий в `main.py` и существующие тесты.

Проверенные файлы:

- `src/gradebook.py`
- `src/student.py`
- `src/utils.py`
- `main.py`
- `tests/test_gradebook.py`

### Какие проблемы нашел

1. `get_score_by_name()` молча возвращает `None`, если студент не найден.
   - Файл: `src/gradebook.py`
   - Проблема: остальные методы для отсутствующего студента выбрасывают `ValueError`, а поиск по имени возвращает `None`. Вызывающий код может принять `None` за допустимый результат.

2. Поиск по имени неоднозначен при одинаковых именах.
   - Файл: `src/gradebook.py`
   - Проблема: если добавить двух студентов с одинаковым `name`, метод вернет оценку первого найденного студента без ошибки.

3. `show_all_students()` отдает наружу изменяемые объекты `Student`.
   - Файл: `src/gradebook.py`
   - Проблема: внешний код может изменить `score`, `name` или `student_id` напрямую, обходя методы `GradeBook`.

4. Нет runtime-валидации входных данных.
   - Файлы: `src/gradebook.py`, `src/student.py`
   - Проблема: можно добавить пустое имя, `None`, строковую оценку, отрицательную оценку или значение вне ожидаемого диапазона. Type hints во время выполнения это не запрещают.

5. `print_students()` корректно работает с пустым списком, но не с пустым генератором.
   - Файл: `src/utils.py`
   - Проблема: пустой генератор truthy, поэтому функция ничего не напечатает вместо сообщения `Журнал пуст`.

6. Тесты не запускаются некоторыми стандартными командами из корня проекта.
   - Файл: `tests/test_gradebook.py`
   - Проблема: `python3 -m unittest discover -v` нашел 0 тестов, а прямой запуск `python3 tests/test_gradebook.py` падает с `ModuleNotFoundError: No module named 'src'`. С `PYTHONPATH=.` тесты проходят.

### Что рекомендуется исправить

- Сделать поведение `get_score_by_name()` явным: выбрасывать `ValueError` или документированно возвращать `None`.
- Решить, допустимы ли одинаковые имена. Если нет, запрещать их при добавлении; если да, возвращать список совпадений или рекомендовать поиск по `id`.
- Добавить валидацию `student_id`, `name`, `score`, включая тип и диапазон оценки.
- Не отдавать наружу изменяемое внутреннее состояние напрямую: возвращать копии, read-only DTO или явно принять этот риск.
- Починить запуск тестов из корня проекта.
- Добавить тесты на edge cases: неизвестное имя, пустой журнал, дубли имен, некорректные оценки и внешнюю мутацию `Student`.

## Tests reviewer result

### Что проверял агент

`tests_reviewer` выполнил read-only проверку тестов:

- покрытие основных функций `GradeBook`;
- ошибочные сценарии;
- пустые значения и пустой журнал;
- запуск через `unittest` и `pytest`;
- недостающие тестовые сценарии.

### Какие проблемы нашел

1. Текущие тесты проходят:
   - `python3 -B -m unittest discover -s tests -v`: 7 тестов, OK.
   - `python3 -B -m pytest -q`: 7 passed.

2. Покрытие `test_add_student` поверхностное.
   - Файл: `tests/test_gradebook.py`
   - Проблема: тест проверяет только количество студентов, но не проверяет `student_id`, `name`, `score` созданного объекта.

3. Не хватает тестов на ошибки.
   - Файлы: `tests/test_gradebook.py`, `src/gradebook.py`
   - Не покрыты:
     - `get_score_by_id` для неизвестного `student_id`;
     - `delete_student` для неизвестного `student_id`;
     - `get_score_by_name` для неизвестного имени.

4. Не хватает тестов на пустые значения и пустой журнал.
   - Файлы: `tests/test_gradebook.py`, `src/gradebook.py`, `src/utils.py`
   - Не покрыты:
     - `show_all_students()` на пустом журнале;
     - `print_students([])`;
     - пустое имя;
     - `None` в `student_id`, `name`, `score`;
     - отрицательная или нечисловая оценка.

5. Нет тестов для функций вывода.
   - Файл: `src/utils.py`
   - Не проверяются `print_student()` и `print_students()`, включая ветку `Журнал пуст`.

6. Нет отдельного теста для `Student`.
   - Файл: `src/student.py`
   - Риск низкий, потому что это dataclass, но создание и поля можно зафиксировать тестом.

### Что рекомендуется исправить

- Добавить тесты на неизвестный id для `get_score_by_id()` и `delete_student()`.
- Добавить тест на `get_score_by_name()` с отсутствующим именем и сначала определить ожидаемое поведение: `None` или `ValueError`.
- Усилить `test_add_student`: проверять `student_id`, `name`, `score`, а также значение по умолчанию `score=0`.
- Добавить тесты пустого журнала и пустого списка для `print_students()` с перехватом stdout.
- Определить правила для невалидных значений и покрыть их тестами.

## Style reviewer result

### Что проверял агент

`style_reviewer` выполнил read-only проверку читаемости и стиля Python-кода.

Проверенные файлы:

- `main.py`
- `src/gradebook.py`
- `src/student.py`
- `src/utils.py`
- `tests/test_gradebook.py`

### Какие проблемы нашел

Серьезных проблем читаемости или чрезмерно сложных функций не найдено. Код небольшой, функции короткие, названия в целом понятные.

Найдены стилевые проблемы, которые могут мешать сопровождению:

1. Wildcard-import.
   - Файл: `main.py`
   - Проблема: `from src.utils import *` скрывает, какие функции реально импортируются.

2. Неиспользуемый импорт.
   - Файл: `src/utils.py`
   - Проблема: импорт `Student` не используется.

3. Повторяется проверка существования студента по `student_id`.
   - Файл: `src/gradebook.py`
   - Проблема: одинаковая проверка встречается в нескольких методах. При росте класса это может привести к расхождению поведения.

4. Название `students` не полностью отражает структуру данных.
   - Файл: `src/gradebook.py`
   - Проблема: поле фактически является словарем по id, поэтому `students_by_id` было бы точнее.

5. Форматирование не полностью соответствует PEP 8.
   - Файлы: `src/gradebook.py`, `src/student.py`, `src/utils.py`, `tests/test_gradebook.py`
   - Проблема: в нескольких местах не хватает пустых строк между импортами и top-level class/function.

### Что рекомендуется исправить

- Заменить wildcard-import в `main.py` на явный импорт `print_students`.
- Удалить неиспользуемый импорт из `src/utils.py` или добавить полезные аннотации типов.
- При ближайшем рефакторинге переименовать `students` в `students_by_id`.
- Вынести получение студента по id в приватный helper, если класс будет расширяться.
- Прогнать автоформаттер вроде `black` или вручную привести пустые строки к PEP 8.

## Общий вывод

Все четыре subagents выполнили отдельные read-only проверки и не изменяли код проекта.

Критичных проблем в основном happy-path сценарии не найдено: базовые тесты проходят, запрещенные временные артефакты не отслеживаются git, функции небольшие и читаемые.

Главные риски проекта сейчас:

1. Непоследовательное поведение `get_score_by_name()`: неизвестное имя возвращает `None`, а не явную ошибку.
2. Неоднозначность поиска по имени при одинаковых именах.
3. Отсутствие runtime-валидации входных данных.
4. Недостаточное тестовое покрытие edge cases и функций вывода.
5. Неполный `.gitignore` для `*.pyc`, `.pytest_cache/` и типовых временных файлов.
6. Неопределенное состояние локальных конфигов `.codex/` и удаленных tracked файлов `REVIEW_REPORT.md`, `codex-agents.json`.

Рекомендуемый порядок исправлений:

1. Определить контракт `get_score_by_name()` и покрыть его тестами.
2. Добавить тесты на ошибки, пустой журнал, дефолтный `score=0` и функции вывода.
3. Ввести правила валидации данных, если проект должен защищаться от некорректных значений.
4. Дополнить `.gitignore` и решить судьбу `.codex/`, `REVIEW_REPORT.md`, `codex-agents.json`.
5. После функциональных решений выполнить небольшой style cleanup: явные импорты, удаление неиспользуемого импорта и форматирование.
